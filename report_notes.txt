BÖLÜM 1: DOCKER + QEMU CI/CD (OTOMATİK DERLEME VE BOOT DOĞRULAMA)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Gömülü sistem projelerinde her değişikliğin derlenip test edilmesi kritiktir.
Bu projede GitHub Actions üzerinde izole bir Docker container kullanılarak
tam otomatik CI/CD pipeline'ı kurulmuştur.

Pipiline Akışı:
  1. git push (tetikleyici) → GitHub Actions Runner
  2. Docker Build: Cross-compiler (aarch64-linux-gnu) ve QEMU içeren imaj
  3. Derleme: 'make all' ile Kernel modülü, IPC uygulamaları (C) derlenir.
  4. RootFS: BusyBox ve derlenen araçlar kullanılarak ext4 rootfs üretilir.
  5. QEMU Smoke Test: QEMU 20 saniyelik timeout ile başlatılır.
     - "VFS: Mounted root" (Dosya sistemi doğrulaması)
     - "TELEMETRY SYSTEM AUTO-START" (Başlangıç scripti doğrulaması)
     Kontrollerinden geçerse test PASS olur ve Artifacts yüklenir.

Docker Kullanım Sebebi: CI sunucusunun bağımlılıklarından kurtulup hem lokalde 
hem de CI'da tamamen aynı ortamın (reproducibility) garanti edilmesidir.

BÖLÜM 2: WEB TABANLI TELEMETRİ İZLEME (BUSYBOX HTTPD + CGI)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Geleneksel web sunucuları (Apache/Nginx) yerine sadece ~20KB boyutunda olan
ve RootFS içerisinde hazır gelen BusyBox httpd kullanılmıştır.

Veri Akış Mimarisi:
  sensor_reader → /dev/shm (IPC) → logger_app → /tmp/telemetry.log
  Browser → httpd:8080 → cgi-bin/telemetry.cgi → tail -10 log

Teknik Detaylar:
  - QEMU Port Forwarding: Host'un 8080 portu, QEMU içerisindeki 8080'e 
    yönlendirilir (-netdev user,hostfwd=tcp::8080-:8080).
  - CGI (Common Gateway Interface): telemetry.cgi, log dosyasının son 10
    satırını okur ve "text/plain" olarak döndürür.
  - C Tabanlı Formatlama: Logger uygulaması (logger_app) performansı
    etkilemeden ham sensör değerlerini doğrudan Floating Point (ondalıklı) 
    yapıya (örn: 23.48°C) dönüştürerek CSV'ye yazar.
  - Frontend: Sadece Vanilla JS kullanılarak (framework olmadan) her saniye
    CGI'ye HTTP GET isteği (polling) atılır ve DOM güncellenir.

BÖLÜM 3: YAPAY ZEKA DESTEKLİ OTOMATİK LOG ANALİZİ (GITHUB MODELS API)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Sistemin ürettiği sensör verilerini anlamlandırmak için GitHub Models 
(Azure OpenAI) altyapısı kullanılarak "gpt-4o-mini" AI modeli entegre edilmiştir.

Güvenlik (API Key) ve Mimari:
  1. Gömülü cihaz internete kapalı/kısıtlı olduğu için AI isteği Backend'de 
     değil, Frontend (Kullanıcı Tarayıcısı) üzerinden yapılır.
  2. GITHUB_KEY, sistem derlenirken ana makinedeki .env dosyasından okunur
     ve güvenli bir şekilde '/cgi-bin/get_api_key.cgi' içine enjekte edilir.
  3. Dashboard açıldığında bu CGI'dan token'ı çeker. Tarayıcı önbellek
     sorununu aşmak için 'cache: no-store' kullanılır.
  4. Eğer CI sunucusunda derlenmişse (token yoksa), tarayıcıda bir prompt
     çıkar ve kullanıcıdan GitHub PAT (Personal Access Token) istenir.

İstek Yapısı (Prompt Engineering):
  Dashboard, son 50 telemetri satırını birleştirerek gpt-4o-mini modeline:
  "You are an AI diagnostic assistant for an embedded Linux telemetry system...
   Highlight any potential anomalies. Keep it under 3-4 sentences."
  komutuyla Bearer Authorization kullanarak gönderir. Modelden dönen analiz,
  dinamik olarak ekrandaki yeşil panele basılır.

BÖLÜM 4: QEMU AArch64 BOOT VE EXT4 ROOTFS MİMARİSİ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Oluşturulan minimal sistem initramfs yerine gerçek donanım I/O davranışını
simüle etmek için 50MB ext4 raw disk imajı üzerinden boot edilir.

QEMU Başlatma Parametreleri:
  -machine virt -cpu cortex-a72 -m 256M -nographic
  -kernel Image
  -drive file=rootfs.ext4,format=raw,if=virtio
  -append "root=/dev/vda rw console=ttyAMA0 earlycon=pl011,0x09000000 panic=5"

VirtIO Blok Cihazı:
Geleneksel emülasyondaki (IDE/SATA) performans kaybını engellemek için VirtIO
paravirtualization kullanılır. Guest kernel (Linux) istekleri paylaşımlı bir
ring buffer'a (virtqueue) yazar ve QEMU bu istekleri doğrudan host dosya 
sistemine pread/pwrite ile iletir. Kernel'da cihaz /dev/vda olarak görünür.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BÖLÜM 5: ARMv8-A MİNİMAL BUSYBOX ROOTFS YAPISI
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
QEMU üzerinde AArch64 için tamamen kendi kendine yeten (statically linked) 
bir BusyBox RootFS kurulmuştur.

FHS (Filesystem Hierarchy Standard) ve Init:
  - /bin, /sbin, /etc, /dev, /proc, /sys yapıları betik ile oluşturulur.
  - /etc/inittab: Sysinit ve ttyAMA0 (UART konsol) respawn tanımlarını içerir.
  - /etc/init.d/rcS: devtmpfs, proc, sysfs, tmpfs gibi sanal dosya 
    sistemlerini mount eder ve telemetri sürecini (IPC + Web) başlatır.
  
BusyBox'ın Statik Derlenmesi (CONFIG_STATIC=y):
Dinamik libc/musl bağımlılıklarını ortadan kaldırarak rootfs boyutunu 
küçültür ve kernel panic hatalarının önüne geçer.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BÖLÜM 6: DEVICE TREE (DTS) VE KERNEL DRIVER BINDING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
x86 sistemlerdeki PCI taramasının aksine, ARM mimarisinde donanımın yeri
kernel'a Device Tree Blob (DTB) ile bildirilir.

DTS Hiyerarşisi ve Eşleşme (Binding):
  - Sensörler (ör. BMP280, MPU6050) I2C controller node'unun altında 
    tanımlanır (reg = <0x76>).
  - "compatible" stringi ("bosch,bmp280") en kritik özelliktir.
  - Kernel boot aşamasında DTB'yi parse edip of_device_id tablolarını 
    tarayarak doğru kernel driver'ını (drivers/iio/pressure/bmp280-i2c.c) bulur.
  - Eşleşme sağlanırsa driver->probe() fonksiyonu çağrılır ve /sys/bus/iio
    altında donanım kullanıcı alanına (userspace) açılır.

Not: QEMU "virt" machine gerçek bir I2C controller sunmadığı için fiziksel
sensör entegrasyonu gerçek donanımda (ör. Raspberry Pi) doğrulanır.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BÖLÜM 7: LINUX I2C KARAKTER SÜRÜCÜSÜ (DRIVER ANALİZİ)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Kernel modülünün yaşam döngüsü iki ana fonksiyona dayanır:
  - probe(): Device Tree üzerinden eşleşme sağlandığında çağrılır. Bellek 
    tahsisi, interrupt ayarı ve /dev düğümünün (node) oluşturulmasını sağlar.
  - remove(): Modül sistemden kaldırıldığında çağrılır, tahsis edilen
    bellekleri güvenle serbest bırakır (memory leak engellenir).

Kullanıcı alanına veri aktarımı:
Kernel Space ile User Space farklı sanal adres alanları kullanır. Güvenlik 
zafiyetleri (ör. page fault, yetkisiz bellek erişimi) yaşanmaması için 
işaretçiler doğrudan dereference edilemez; veri aktarımı 'copy_to_user()' 
ve 'copy_from_user()' makrolarıyla güvenli şekilde sağlanır.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BÖLÜM 8: POSIX SHARED MEMORY IPC MİMARİSİ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
İki bağımsız süreç ('sensor_reader' ve 'logger_app') arasında yüksek frekanslı 
veri aktarımı için fiziksel bellekte (RAM) ortak bir bölge kullanılır.

Sistem Çağrıları (Syscalls):
  - shm_open(): /dev/shm altında bellek tabanlı dosya oluşturur.
  - ftruncate(): Bellek segmentinin boyutunu ayarlar.
  - mmap(): File descriptor'ı sürecin sanal adres alanına eşler.

Senkronizasyon:
Ring buffer, overflow/underflow durumlarını engellemek için iki adet
POSIX Semaphore (sem_w: boş slot sayısı, sem_r: dolu slot sayısı) ile yönetilir.

BÖLÜM 9: POWER RESILIENCE (GÜÇ KESİNTİSİ DAYANIKLILIĞI)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Gömülü sistemin elektrik kesintilerine dayanıklı dosya yazma yeteneği 
'filesync' modülü ve logger üzerinde sağlanmıştır.

Kritik Özellikler:
  - O_DSYNC Flag: Verilerin Kernel Page Cache'te beklemeden doğrudan fiziksel
    disk cihazına (Storage) gitmesini zorunlu kılar.
  - Atomic Write Pattern (Güvenli Kopyalama): Dosya geçici bir isme 
    (.sync_tmp) yazılır, fdatasync() ile kalıcı hale getirilir, rename() ile 
    hedefe taşınır ve dizin sync_directory() ile güncellenir. Yarım yazılmış 
    (corrupt) dosya ihtimali sıfıra indirgenir.

2. ÖRNEK KOD: CROSS COMPILATION VE IPC SİSTEMİ (Aşama 1 & 4)
---------------------------------------------------------
Shared Memory (Ring Buffer) üzerinden haberleşen süreçler için tanımlanan C yapısı:

```c
// source_code/ipc/includes/ipc.h (Özet)
typedef struct s_telemetry {
    uint32_t seq;
    int32_t  temp;
    int32_t  pres;
    int32_t  hum;
    int16_t  accel_x;
    int16_t  accel_y;
    int16_t  accel_z;
    uint64_t timestamp;
} t_telemetry;

typedef struct s_shm_ring {
    volatile uint64_t write_idx;
    volatile uint64_t read_idx;
    volatile int      shutdown;
    sem_t             sem_w; // Boş slot sayacı
    sem_t             sem_r; // Dolu slot sayacı
    t_telemetry       slots[RING_SIZE];
} t_shm_ring;
```
Derleme: `aarch64-linux-gnu-gcc -Wall -O2 -o sensor_reader sensor_reader.c -lpthread -lrt`

3. ÖRNEK KOD: MINIMAL ROOTFS OLUŞTURMA (Aşama 2)
---------------------------------------------------------
Bash script ile FHS dizin yapısının ve BusyBox init'in ayarlanması:

```bash
# rootfs/build_rootfs.sh (Özet)
mkdir -p rootfs_staging/{bin,sbin,etc,proc,sys,dev,usr/bin,usr/sbin}

# Init Konfigürasyonu
cat << 'EOF' > rootfs_staging/etc/inittab
::sysinit:/etc/init.d/rcS
::respawn:-/bin/sh
ttyAMA0::respawn:-/bin/sh
EOF

# cpio ile initramfs paketleme
cd rootfs_staging
find . -print0 | cpio --null -ov --format=newc | gzip -9 > ../rootfs.cpio.gz
```

6. ÖRNEK KOD: KERNEL CHARACTER DRIVER ANALİZİ (Aşama 6)
---------------------------------------------------------
File operations yapısı ve Kernel-Space'ten User-Space'e veri kopyalama mantığı:

```c
// source_code/driver/telemetry_sensor.c (Özet)
static ssize_t sensor_read(struct file *file, char __user *user_buf, size_t count, loff_t *ppos)
{
    // ... Sensör verisi (buf) oluşturulur ...
    // Doğrudan pointer kullanmak yerine güvenli aktarım (page fault önlemi)
    if (copy_to_user(user_buf, buf, len)) {
        return -EFAULT;
    }
    *ppos += len;
    return len;
}

static const struct file_operations sensor_fops = {
    .owner   = THIS_MODULE,
    .read    = sensor_read,
    .open    = sensor_open,
    .release = sensor_release,
};
```
